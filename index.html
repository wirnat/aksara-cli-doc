Aksara CLI

Speed Up your coding process with Smart Code Generator.

  • Doc for Version: 0.0.3
  • Author: Wirnat
  • Created: 9 October, 2022
  • Last Updated: 12 October, 2022

Aksara CLI is code generator that used to build repeatable code depend on the model. For example, If you work using architecture layer this tool will useful to help build your CRUD(create update delete) on repository, delivery, or usecase layer, even mocking an service. So you can just focus to the business logic on the project.

Installation

go install github.com/wirnat/aksara-cli@v0.0.3

Quick Example

Let's practices with a simple case, assume we want to create a CRUD for modules company and branch, but only in repository layer.

First, prepare the models in model directory:

            
package model

type Branch struct {
  BaseModel
  CompanyID   int64   `json:"company_id"`
  Name        string  `json:"name"`
  Description *string `json:"description"`
}

/*
  @model Branch
  @service branch
*/
            
       
          
package model

type Company struct {
	BaseModel
	Name string `json:"name"`
}

/*
	@model Company
	@service company
*/

          

            
package model

import (
	"time"
)

type BaseModel struct {
	ID        int64      `json:"id"`
	UUID      string     `json:"uuid"`
	CreatedAt time.Time  `json:"created_at"`
	UpdatedAt time.Time  `json:"updated_at"`
	DeletedAt *time.Time `json:"deleted_at"`
}

Model work as backbone in this generator, if @model comment is found in model, it will execute the model to the generator.

Then, create a json file with the name simple_crud.json

You can use other name of json file if you want.

            
              {
                "key": "ᬅᬓ᭄ᬱᬭ",
                "model_path": "model",
                "result_path": "modules",
                "module_name": "github.com/wirnat/aksara-cli",
                "module_traits": [
                  {
                    "name": "~service~ repository interfaces",
                    "dir": "~result_path~/~service~/~service~_repository",
                    "template": "template/clean_architecture/repository_interfaces.text",
                    "file_name": "~service~.go",
                    "meta": {
                      "import_request": "~module_name~/~result_path~/~service~/~service~_request"
                    }
                  },
                  {
                    "name": "~service~ fetch param",
                    "dir": "~result_path~/~service~/~service~_request",
                    "template": "template/clean_architecture/request_param.text",
                    "file_name": "~service~_request.go",
                    "meta": {}
                  },
                ]
              }
This file will work as a configurator, generator and aggregator, so you must assign:
key, original key (ᬅᬓ᭄ᬱᬭ)
model_path, model directory
result_path, the location of result
module_name, the name of your project in go mod
module_traits is a file generator, it will generate file base on the template which assign in template.
  name, is the name of action
  dir, the location of generated file (it will auto generate folder)
  file_name, the name of generated file
  meta, is a custom meta variable that can called in template.

Module Trait will executed when the generator read the model that has @model meta

But, what is ~service~ , @model , @service, etc ?

Aksara CLI has default meta and can called in generator json file or in template file.

Json key of the configurator is a default meta (except module_traits). In json file you can use ~meta name~ to make your config more dynamic and readable. Inside the module_traits, we can use meta which declared in model file like ~service~


Declared meta on model

Now let's create the template, at module_traits in json file we have declared 2 generate file, so we need create 2 template too.

Template for interface repository file: 

                
package {{.Meta.package_repo}}

import (
  "context"
  "{{.ModuleName}}/{{.ModelPath}}"
)

type {{.Model}}Repository interface {
  {{.Model}}RepositoryStore
  {{.Model}}RepositoryUpdate
  {{.Model}}RepositoryGet
  {{.Model}}RepositoryFetch
}

type {{.Model}}RepositoryStore interface {
  Store(ctx context.Context, branch *model.{{.Model}}) error
}

type {{.Model}}RepositoryUpdate interface {
  Update(ctx context.Context, branch *model.{{.Model}}) error
}

type {{.Model}}RepositoryGet interface {
  Get(ctx context.Context, uuid string) (model.{{.Model}}, error)
}

type {{.Model}}RepositoryFetch interface {
  Fetch(ctx context.Context) ([]model.{{.Model}}, error)
}

Template for repository implementation file: 

                
package {{.Meta.package_gorm}}

import (
  "context"
  "{{.ModuleName}}/{{.ModelPath}}"
  "gorm.io/gorm"
)

type {{.Meta.service}}Gorm struct {
  db gorm.DB
}

func New{{.Model}}Gorm(db gorm.DB) *{{.Meta.service}}Gorm {
  return &{{.Meta.service}}Gorm{db: db}
}

func (b {{.Meta.service}}Gorm) Store(ctx context.Context, {{.Meta.service}} *model.{{.Model}}) error {
  return b.db.Create({{.Meta.service}}).Error
}

func (b {{.Meta.service}}Gorm) Update(ctx context.Context, {{.Meta.service}} *model.{{.Model}}) error {
  return b.db.Updates({{.Meta.service}}).Error
}

func (b {{.Meta.service}}Gorm) Get(ctx context.Context, uuid string) (r model.{{.Model}}, err error) {
  err = b.db.Where("uuid=?", uuid).First(&r).Error
  return
}

func (b {{.Meta.service}}Gorm) Fetch(ctx context.Context) (r []model.{{.Model}}, err error) {
  err = b.db.First(&r).Error
  return
}
Double curly bracket will call variable which declared by default in generator such as {{.Model}}, json file variable, or meta that's setted in model or json file.

Execute the generator by following command,aksara-cli generate [your json file] : 


aksara-cli generate simple_crud.json

The result is a complete set of folders with files formed by the template

How it's work?

Aksara-cli works depend on template, model, and json file. On start process generator will read json file and scan the model directory to find model that has @model in model file, then module_traits in json file will execute template and, meta variable base on scanned model will run on this process.

Meta

Meta is a variable that used for json and template file. Meta can be set from model file or json file.

Model Meta

To add meta from model, you can write comment in model file (any where in file) and add @ as a prefix of meta key followed by meta value

            
/*
  @model Company
  @model_snake_case company
*/
Consider if you want to use variable base on model (scanned model turn), add meta from model is the best choice

Model Field Meta

By default, every fields of model has stored to model_fields meta and can be used in template file,
but you also can add meta to every field when template is getting complicated.

  
    type Branch struct {
      BaseModel
      CompanyID   int64   `json:"company_id"` //@meta validate:true
      Name        string  `json:"name"` //@meta validate:true
      Description *string `json:"description"`
    }
Json Meta

You also can add meta from json file, by adding meta in module_traits, but keep in your mind meta must be unique

             
{
  "key": "ᬅᬓ᭄ᬱᬭ",
  "model_path": "model",
  "result_path": "modules",
  "module_name": "github.com/wirnat/aksara-cli",
  "module_traits": [
    {
      "name": "~service~ repository interfaces",
      "dir": "~result_path~/~service~/~service~_repository",
      "template": "template/clean_architecture/repository_interfaces.text",
      "file_name": "~service~.go",
      "meta": {
        "app_name":"extraction" 
      }
    }
  ]
}
Meta Usage

In json file you can call meta using ~[meta-name]~, in template you can call meta using {{.[meta-name]}}

Default Meta
  • Model - Name of scanned model
  • ModelFields - List of model fields, every field contains:
    • Json - string
    • Name - string
    • Type - string
    • IsPtr - bool
    • Meta - map[string]interface{}
  • ModuleName - Declared Module Name
  • ResultPath - Declared result path
  • ModelPath - Declared model path

Template

Main former of this generator is The Template, if json file serve as directory and architecture builder, template is the code builder. Template is compiled when all meta, variable, and config in json and model file has been compiled.

Template will be compiled using default template from Template Go, so you can use directive, variable call, and all feature of The Golang Template.

Here some template examples:

Loop and meta: 
              
package {{.Meta.service}}_gorm

import (
	"{{.ModuleName}}/{{.ResultPath}}/{{.Meta.service}}/{{.Meta.service}}_request"
	"gorm.io/gorm"
)

func filter(db *gorm.DB, param {{.Meta.service}}_request.{{.Model}}Param) {
	{{- range $m:=.ModelFields }}
	if param.{{$m.Name}} != nil {
    db = db.Where("{{$m.Name}}", *param.{{$m.Name -}})
    }
    {{- end}}
}
